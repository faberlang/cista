# Delivery Spec: Phase A — Local library store loop

**Status**: shipped — Phase A delivery history (verified in current cista tests)
**Goal**: `docs/factory/cista-package-store/goal.md`  
**Unit**: Phase A — finish local library store (close the loop)  
**Repos**: cista, faber, examples (cross-repo file contract; no crate deps)

## Historical Interpreted Unit

Close the local library package loop. This phase is shipped; the following
requirements and validation remain as delivery history: install a Rust lib into
`$CISTAE_HOME`, rewrite project `faber.lock` with absolute paths, and make
`faber check` /
`faber build` consume those locked interface (and validate artifact) paths
without knowing about cista or `CISTAE_HOME`.

## Normalized Spec

### In scope

1. **cista** inspect / package list / package show / package files / remove
2. **cista install --path** + `--project`/cwd project-root rule + `faber.lock` rewrite
3. **faber** `[dependencies]` in `faber.toml` + `faber.lock` reader
4. **faber** library resolution from locked `interface_root` for non-bundled providers
5. **examples/cista-lab/demo** consumer fixture
6. Validation that locked artifact paths exist when present

### Out of scope

- Phase B Norma packaging
- Bins, `cista run`, meta, registry
- Shared Rust crate between faber and cista
- Full rlib cargo linking (interface codegen from Faber bodies remains valid for mathesis)

### Invariants

- No faber↔cista crate dependency
- faber does not read `CISTAE_HOME` or call cista
- Store discovery is cista-owned
- Exact direct deps only

## Repo-Aware Baseline

| Surface | Today |

## Stage Graph

1. **A1 cista store ops** — list/show/files/remove/inspect against store layout  
2. **A2 cista lock rewrite** — project root rule + write `faber.lock`  
3. **A3 faber lock consume** — manifest deps + lock parse + resolver  
4. **A4 demo fixture** — mathesis consumer end-to-end  
5. **A5 verify** — unit/integration tests + manual install/build path  

## Implementation Work

### cista

- `src/faber_lock.rs` — lock record TOML read/write (package-manager side)
- `src/project_manifest.rs` — minimal `faber.toml` `[dependencies]` parse for lock rewrite
- `InstallArgs.project: Option<PathBuf>`
- install: resolve project root; rewrite lock after successful install
- `package` list/show/files; `remove`; `inspect` real implementations
- shared store walk helpers

### faber

- `FaberManifest.dependencies: BTreeMap<String, String>`
- `src/package/lockfile.rs` — `faber.lock` parse + path validation
- `LibraryResolver` locked-package map (`interface_root` layout: `<root>/<module>.fab`)
- `library_resolver_for_package(config, package_root)` loads lock + deps
- Diagnostics for missing lock entries / missing paths / undeclared providers

### examples

- `cista-lab/demo/{faber.toml,src/main.fab}` (lock generated by install)

## Checkpoints And Gates

- **PASS**: install mathesis with `--project demo` writes lock; `faber check`/`build` on demo succeeds using locked interfaces; package list/show/remove work; no cross-crate deps  
- **Release**: defer-release (early plumbing; not a user release)  
- **Batching**: one Phase A batch  

## Validation

```bash
# cista
cargo test -p cista
cargo run -p cista -- package list --store /tmp/cistae-test
cargo run -p cista -- install --path ../examples/cista-lab/source/mathesis \
  --target-language rust --store /tmp/cistae-test \
  --project ../examples/cista-lab/demo

# faber
cargo test -p faber
cargo run -p faber -- check ../examples/cista-lab/demo
cargo run -p faber -- build ../examples/cista-lab/demo
```
